# Turabi Labs — Customer Ordering Website (Phase 2)

Next.js 14 (App Router) + Supabase implementation of the customer-facing ordering
flow from the Phase 2 brief: QR/link → table → mobile number → menu → cart →
checkout → order confirmation with a permanent, live-updating status link.

## Setup

```bash
npm install
cp .env.local.example .env.local   # fill in your Supabase URL + anon key
npm run dev
```

Visit a tenant like: `http://localhost:3000/order/demo-restaurant?table=12`

## Expected Supabase tables

This app assumes the Phase 1 schema, with these table/column names:

| Table         | Key columns used here                                                                 |
|---------------|-----------------------------------------------------------------------------------------|
| `restaurants` | `id, name, logo_url, subdomain, subscription_status`                                    |
| `tables`      | `id, tenant_id, table_number`                                                           |
| `categories`  | `id, tenant_id, name, sort_order`                                                        |
| `menu_items`  | `id, tenant_id, category_id, name, description, price, image_url, is_available`         |
| `customers`   | `id, tenant_id, mobile_number` — needs a **unique constraint on `(tenant_id, mobile_number)`** for the upsert in `placeOrder()` to work |
| `orders`      | `id, tenant_id, table_id, customer_id, order_code, status, payment_method, payment_status, total_amount, created_at` |
| `order_items` | `id, order_id, menu_item_id, quantity, price_at_order`                                   |

If your Phase 1 SQL used different table or column names, update `lib/queries.ts` —
it's the only file that talks to Supabase directly.

`order_code` is expected to be generated by the Postgres trigger from Phase 1
(format `ORD-YYYYMMDD-XXXXXX`) — this app never generates it client-side.

## How the refresh-resilience requirements are implemented

| Requirement | Where |
|---|---|
| Table number survives refresh via the URL | `?table=` query param threaded through every route (`app/order/[tenant]/**`) |
| Mobile number remembered on-device | `lib/storage.ts` → `localStorage`, read/written in `OnboardingFlow.tsx` and `CheckoutForm.tsx` |
| Cart persisted and restored | `lib/cart-context.tsx` hydrates from `localStorage` on mount, syncs on every change, clears only after a successful order |
| Order tracking survives closed tab / different device | `app/order/track/[orderCode]/page.tsx` — looks the order up **only** by `order_code` against Supabase, never from local state |

## Where later phases plug in

- **Phase 4 (Realtime):** `components/OrderTracker.tsx` currently polls Supabase
  every 5s (`POLL_INTERVAL_MS`). Swap the `setInterval` loop for a
  `supabase.channel(...).on('postgres_changes', ...)` subscription.
- **Phase 5 (Razorpay):** `components/CheckoutForm.tsx` has a `TODO` where the
  Razorpay checkout modal should open before the order is placed when
  `method === 'online'`.
- **Phase 6 (Invoices):** the disabled "View invoice" button in
  `OrderTracker.tsx` becomes a real download link once invoice generation and
  storage are wired up.

## File map

```
app/
  layout.tsx                          root layout, fonts
  page.tsx                            placeholder home page
  order/
    [tenant]/
      layout.tsx                      wraps routes in CartProvider
      page.tsx                        table confirm + mobile entry (steps 1-3)
      menu/page.tsx                   digital menu (step 4)
      cart/page.tsx                   cart (step 5)
      checkout/page.tsx               payment method + place order (steps 6-7)
    track/[orderCode]/page.tsx        permanent live status link (step 8)
components/                           UI pieces (all documented inline)
lib/
  supabase.ts                         browser client
  queries.ts                          all Supabase reads/writes
  storage.ts                          localStorage helpers
  cart-context.tsx                    cart state + persistence
  types.ts                            shared types matching the schema
  format.ts                           price/time formatting
```
